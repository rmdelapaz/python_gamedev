## Conventions / things to remember

- **Filesystem tool > computer use tool** for all WSL file ops in this project
- **Prefer `edit_file` over `write_file`** for existing files (less rewrite risk; `session.md` is the explicit exception ŌĆö it gets fully overwritten every milestone)
- **Default to region-scoped reads (chat 27 added; effective chat 28+).** For lesson HTML, precedent lesson HTML, and precedent SVG, use `head` / `tail` / `view_range` to read only the relevant region ŌĆö not the full file. Build chats were consuming 30ŌĆō50% more input tokens than necessary by full-reading files when only ~50 lines were relevant. Triggers: target lesson ŌåÆ grep for `svg-wrapper` and the canvas-wrapper anchor first, then `view_range` read of the mermaid-and-canvas-wrapper region only; precedent lesson ŌåÆ same; precedent SVG ŌåÆ header comment (the build narrative) + `tail 5` to confirm closing `</svg>`, body skipped unless visual-language reuse is genuinely needed. Full-file reads should be a deliberate escalation, not the default.
- **Keep `status.md` chat-N sections concise (chat 27 added; effective chat 28+).** Aim for ~1.5 KB per chat-N section, not the 7 KB chat-25/26/27 sections grew to. The walkthrough doesn't need to be exhaustive ŌĆö it needs to be enough that an interrupted chat's recovery can pick up state. The artifacts on disk (SVG, lesson HTML, catalog JSON, this `status.md` itself once trimmed) ARE the source of truth; the narrative is annotation, not duplication. Trim ruthlessly: SVG build details that are visible by reading the SVG header comment don't need to be repeated; getBBox numbers tighter than ~20 px right margin are interesting and should be recorded, anything healthier is just "clean." Save-at-milestones (chat-23 discipline) becomes more powerful when each milestone save is small.
- **Intermediate catalog status `built_unwired_chat_N` (chat 27 added; effective chat 29+ ŌĆö the next build chat).** Between SVG file landing on disk and wire+catalog finalize, flip the lesson's catalog entry to `built_unwired_chat_N` immediately after the SVG file is written. If compaction hits between SVG and wire ŌĆö the dominant failure mode (5/5 interrupted-artifact cases hit exactly this window) ŌĆö the catalog records the build automatically and recovery is even cheaper than chat 27. Wire completion then flips to `built chat N` as today. New status flow: `needs_new_svg` ŌåÆ `built_unwired_chat_N` ŌåÆ `built chat N`. Schema note added to `support/phase7-canvas-fallback-catalog.json`.
- **Slim chat-N+1 kickoff prompt (chat 27 added; effective chat 28+).** Target ~500 bytes for build-chat kickoffs; modestly longer for render-check kickoffs (the 5 verification points and visual-gate content list need explicit listing). The current 3 KB kickoffs duplicate `session.md` + `todo.md` content; the working files carry the convention details. Kickoff prompt structure: project root + UNC-path reminder; one sentence pointing to `session.md` as resume pointer; one sentence describing the chat-N task; that's it. Everything else lives in `session.md` / `todo.md` / `next-chat-prompt.md`.
- **Stream `status.md` narrative incrementally per milestone (chat 30 added; effective chat 30+).** Don't bunch the chat-N `status.md` narrative as one block at chat close ŌĆö that's the dominant interruption-loss surface (5 of 6 interrupted chats lost the narrative dump while preserving every other on-disk artifact, including chat 29). Instead, append short prose paragraphs to the chat-N `status.md` section *after each pipeline milestone*: lesson read / `build_concept` refinement / SVG file write / wire / catalog flip / getBBox pre-clear / render-check / etc. Each individual `Filesystem:edit_file` call to `status.md` is small (~150ŌĆō300 bytes) and ordered *away* from the chat-end interruption window. Per-milestone writes ride alongside the artifact saves they describe and use the same chat-23 save-first principle, but at finer granularity than `session.md` (which still gets a single milestone overwrite per save). The chat-close `status.md` write is then just a short wrap-up paragraph. **Chat-N section structure under this convention:**
  ```
  ### Phase 7 progress (date chat N ŌĆö task)

  **IN PROGRESS / CLOSED CLEAN.** [one-line headline summary, written at chat start]

  **Milestone 1 ŌĆö name.** [paragraph appended after milestone 1 completes]

  **Milestone 2 ŌĆö name.** [paragraph appended after milestone 2 completes]

  ŌĆ”

  **Chat close.** [short wrap-up appended at chat close]
  ```
  Trade-off: more `Filesystem:edit_file` calls per chat, but each call is small and the per-call cost is dramatically lower than the chat-end-blowup cost. Effective chat 30+ ŌĆö chat 30 (render-check) is the first test bed; chat 31+ build chats are where the bigger interruption-rate dividend is expected. Discipline applies uniformly to build AND render-check chats so it's automatic per chat type, not something each chat has to switch on. **The chat-N section is built as you go, not at the end.**
- **`audit_lessons.py`'s `svg` column counts inline `<svg>` only.** Phase 6 SVGs (delivered as standalone files in `/images/components/` via `<img>`) show up in the `img` column, not `svg`. To count Phase 6 deployments, sum `img` across lessons that received them.
- **Before writing a new SVG, `grep -l 'svg-wrapper'` the target lesson(s).** Pre-staged `<figure class="svg-wrapper">` markup may already dictate filename, layout, or labels. Existing alt text = spec.
- **`canvas { display: none !important; }` is the established mobile-fallback hide rule.** Inline `display:` styles on `<canvas>` tags would otherwise win on specificity (chat 15 finding ŌĆö 3 of 6 canvases on the chat 13/14 wrapped lessons had inline `display: inline-block`). The `!important` neutralizes any inline override, so when wrapping new canvases for Phase 7 you do NOT need to audit each canvas tag for inline display styles before wrapping. **However, source-level "did the rule exist?" reviews are insufficient ŌĆö render-check at phone width is mandatory before declaring a Phase 7 wrap done.**
- **Warn when conversation is getting long** rather than letting auto-compaction hit
- **Save-first-not-last (chat 23 added).** Doc updates happen at milestone boundaries, not chat end. Stub `todo.md` + `status.md` writes at chat start; incremental updates after each milestone (build / insert / catalog flip / render-check). Reverses the chat-19/21/22 failure pattern (saves were ordered *near* the interruption window).
- **`support/session.md` is the canonical resume pointer (chat 23 added).** First file to read on every chat start, before `todo.md`. Overwritten at every milestone ŌĆö single-paragraph snapshot, not accumulating history. Schema: Chat / Date / Status / Last completed / Next step / Process changes (optional). Keep under ~350 words.
- **Don't add content not present in existing lessons** unless explicitly requested
- **Don't add PetalFawnStudio branding** unless explicitly requested
- **Path: `\\wsl$\Ubuntu\ŌĆ”`** not `\\wsl.localhost\Ubuntu\ŌĆ”`
- All injection scripts in this project follow the same shape: idempotent,
  `--dry-run`, `--no-backup`, `.bak` sibling files, color terminal output, summary
  block at end.
